MetPX is a message switching system for use with Wmo Global Telecommunications System circuits based on TCP/IP. Developed at the Canadian Meteorological Centre of Environment Canada for our own use. MetPX is a work in progress, the development process is being externalized in the hopes of fostering a community around a switching system which all can share and improve together. The system is already production quality for a limited set of features and is in production use at the CMC as the core of our national bulletin switching infrastructure. The software is under continuous, but low-intensity development. Features are carefully being layered on top of existing stable functions.

[ mailing-list: metpx-devel. ] [ Main Project page: Sourceforge ]

WARNING: This software has minimal documentation, and can only be installed by a seasoned linux administrator. A portion of the work done as a condition of the release was to strip the repository of configuration examples, since they, being real, indicated sensitive information. So, one of the items on the TODO list is to add real sample configuration files. As downloaded, without a great deal of configuration, this code will not run in a useful way. The placing of this code on sourceforge does not represent the publication of a finished product, but a work-in-progress.

The switch component is command-line driven, but has a web GUI for monitoring purposes. A knowledge of French is helpful currently, as the socket protocol drivers are written in French (as are their log messages.) Proper localization is a goal, but we don't have time yet.

Protocol support:

  • AM (Canadian proprietary socket protocol)
  • WMO (see manual 386 for the WMO TCP/IP socket protocol.)
  • FTP (for transport, no support for WMO naming yet (but trivial to add.))

features:

  • detailed routing (in production with 30,000 distinct entries in the routing table.)
  • sub-second latency (with 28,000 routing entries.)
  • high speed routing and delivery. (was >300 messages per second, but that was a year ago, and many features which might slow it down have been added, a re-test is in order.)
  • no message size limit.
  • message segmentation (for protocols such as AM & WMO which have message length limits.)
  • duplicate suppression (on send.)
  • Near Future (in development):
    • AFTN support?
    • bulletin collections.
  • planned:
    • use of curl or some such in sender to be able to drive a wide number of file transfer protocols.
    • generalized filter mechanism. (collections will be modified to fit this general mechanism.)
Release of stable versions has not yet been worked out. We will tag a release when it is ready, it will not be ready for a number of months at least! If you are still interested, read on. Best bet is to subscribe to the mailing list and let us know what is stopping you from trying it out, it could inspire us to work on that bit faster to get some collaboration going. There are old tagged releases, but they all have known but undocumented (on the sf.net bug tracking system) bugs.

There are two modules in the project right now, and a third under consideration. Modules of MetPX are named after species of plant which are endangered in Canada (see Species at Risk for more details.)

  • sundew: the WMO switching component.
  • columbo: the web based monitoring component, for both sundew, and the older (not released) PDS.

Platform: Debian Linux (Sarge). Any modern Linux should do. (stock 2.6 or 2.4 with many patches.) Python >2.3
